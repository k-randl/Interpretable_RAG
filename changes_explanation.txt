This file explains the changes made to `src/Interpretable_RAG/generation.py` to resolve errors encountered during the execution of `scripts/run_pipeline.py`.

### 1. `ValueError` in `_get_shapley_values_monte_carlo`

**Error:** `ValueError: Cannot take a larger sample than population when 'replace=False'`

**Reason:**
This error occurred in the `_get_shapley_values_monte_carlo` function. The function has a `sample_size` parameter with a default value of 10. The code attempts to take a random sample of this size from a list of `indices`. However, in some cases, the number of available `indices` was less than 10. This caused `np.random.choice` to fail because it was asked to draw a larger sample than the population size without replacement.

**Change:**
A check has been added to ensure that `sample_size` does not exceed the number of available `indices`. If `sample_size` is larger, it is adjusted to be equal to the number of `indices`. This prevents the `ValueError`.

### 2. `ArrayMemoryError` or `permutation` error in `sample_perturbations`

**Error:** `numpy.core._exceptions._ArrayMemoryError: Unable to allocate 2.00 PiB for an array...` or an error inside `numpy.random.mtrand.RandomState.permutation`.

**Reason:**
This error occurred in the `sample_perturbations` function, which generates combinations of query tokens. The function calculates the total number of possible combinations (`n = 2 ** number_of_tokens`).

Initially, the code tried to create an array of all possible combination indices using `np.arange(1, n-1)`, which caused a memory error for long queries.

A subsequent fix replaced `np.arange` with a direct call to `np.random.choice(n-2, ...)`. However, for very large `n`, the internal implementation of `np.random.choice` (which uses `permutation`) also attempts to allocate a large amount of memory, leading to a similar error.

**Change:**
The `sample_perturbations` function has been rewritten to be more memory-efficient. Instead of using `np.random.choice` for large ranges, the new implementation generates unique random numbers one by one using a loop and `np.random.randint`, adding them to a Python `set` to ensure uniqueness. This method has the same outcomeâ€”a random sample of unique combinationsâ€”but avoids the massive memory allocations that were causing the errors.

### 3. `OverflowError` in `sample_perturbations`

**Error:** `OverflowError: Python int too large to convert to C long`

**Reason:**
This error occurs when a Python integer, which can be arbitrarily large, is converted to a fixed-size C long (a 64-bit integer in NumPy). In the `sample_perturbations` function, the number of possible combinations `n` can exceed the maximum value of a 64-bit integer if the number of items is greater than 63. When the code tries to store `n-1` into a NumPy array with the default integer `dtype`, it causes an overflow.

**Change:**
The NumPy array `sample` is now created with `dtype=object`. This allows the array to hold Python `int` objects, which can handle arbitrarily large integers, thereby preventing the `OverflowError`.
